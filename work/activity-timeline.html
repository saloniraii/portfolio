Unified Agentic Activity Timeline

A redesign of Salesforce's most universal CRM component for a near-future where AI agents generate the majority of timeline entries. Self-initiated, designed to production fidelity, and filed as a patent invention disclosure.

Duration
~3 months
Role
Product Designer
Team
2 Designers + 1 Architect
Invention Disclosure Accepted · Patent Pending

The provocation

Agent activity volume is about to overwhelm the timeline.

Activity timelines are the backbone of every Salesforce cloud - Sales, Service, Marketing, Health, Financial Services. As agents ship across these products, timeline entry volume will increase by an order of magnitude. Two problems become urgent: provenance (who did this?) and prioritization (what needs me?).

22

activity types across Salesforce clouds. No attribution layer distinguishes agent actions from human ones.

Agents already qualify leads, triage cases, send follow-ups, and update records. But the timeline treats their work identically to human work. A rep opening a record with 200 overnight agent actions has no way to identify the 3 that need a decision. We didn't wait for a brief - we started designing when we saw the gap forming.

No provenance

Agent-performed and human-performed activities are visually identical. As agents generate more entries, users lose the ability to assess what's already handled.

No prioritization

A rep managing 50 accounts can't parse a timeline where agents logged 200 actions overnight. No triage layer. No "needs human decision" signal.

No rationale

When an agent sends a follow-up email, the timeline shows "Email sent." Not why it was sent, what data informed the action, or whether a human should intervene.

Interaction model

How attribution, triage, and approval flow together

The redesigned timeline introduces a three-step loop. Each step addresses a specific failure mode of the current chronological log.

Attribution surfaces

Every activity displays author type (agent vs. human) and, for agent entries, the triggering condition. Users scan the timeline knowing immediately what was autonomous.

Triage filters

The "Needs Attention" tab collects entries requiring a human decision - escalations, low-confidence agent outputs, items awaiting approval. Users skip the noise.

User acts

On each triaged item, the user can Approve, Edit & Resend, or Undo the agent action. The timeline updates in real-time and the agent learns from the correction.

Key decisions

Four departures from every CRM timeline on the market

Decision 01

Expose agent rationale inline

Each agent-initiated activity includes a collapsible "why" block: the data signals that triggered the action, confidence level, and a recommended next step for the human. Scenario: agent sends a pricing follow-up - rationale shows 3 objections from Maria's last email matched to approved discount tiers.

Decision 02

Separate "Needs Attention" from timeline

A dedicated triage tab surfaces only items requiring human action: escalated cases, low-confidence agent outputs, overdue approvals. Items leave this tab once acted on. Scenario: agent case triage flagged a sentiment drop - appears in Needs Attention until rep reviews.

Decision 03

NL querying over filter dropdowns

The search bar accepts natural language questions ("average sentiment from emails this week?"), not just keyword filters. Returns structured cards with engagement scores rather than just filtered lists. Handles the analytical use case that filter menus can't.

Decision 04

Cross-cloud provenance badges

Activities display their originating cloud (Sales, Service, Marketing, Commerce, Field Service, Revenue) as a visual badge. A single record's timeline shows the complete customer journey with clear source attribution - not siloed per-cloud views.

The strongest design decision

Approve & Send: keeping humans in the loop without slowing them down

The critical design challenge was making human oversight feel lightweight, not bureaucratic. Every agent-initiated action that affects external contacts (emails, meeting requests, proposal changes) routes through a three-button approval flow. The user sees what the agent did, why it did it, and can approve in one click, edit before sending, or undo entirely. This keeps agent throughput high while preserving human accountability for customer-facing actions.

Process

What we tried, what we rejected, and what shaped the final design

This wasn't a linear process. Several ideas looked promising in isolation but failed under the constraints of the Salesforce platform and enterprise use cases.

Rejected

Inline agent chat thread

We explored showing a conversational thread between the user and agent within each activity. Looked elegant in demos, but collapsed under real data: a single activity could generate 8+ messages of context, burying the timeline. Replaced with the collapsible "rationale" block - one summary paragraph per action.

Stakeholder pushback

"Don't make it feel like surveillance"

Early design reviews flagged that showing agent confidence scores on every entry felt like it was grading the AI. Valid concern. We moved confidence to the detail view (not the list) and only surfaced it prominently when it was low enough to trigger escalation. The "it's fine" case stays quiet.

Platform constraint

Activity Timeline is a shared component

The timeline component is consumed by every Salesforce cloud. Changes to its data model affect every downstream team. This forced us to design an attribution layer that was additive (new metadata fields, new rendering logic) rather than restructuring existing activity types. Backward compatibility was non-negotiable.

Tradeoff accepted

NL search latency vs. accuracy

Natural language querying over activity data introduces 2-3 seconds of latency. We considered pre-computing common queries but the combinatorial space was too large. Accepted the latency and designed a clear loading state with progressive disclosure - show partial results as they resolve.

Impact & ownership

Pre-ship vision work with measurable downstream influence

This project has not shipped as a product. It's speculative design work - a vision for where the timeline needs to go as agents scale. Its value is in setting direction and producing reusable patterns.

Invention disclosure accepted by Salesforce IP team (7 inventors). Covers agent attribution, NL timeline querying, actionable AI summaries, and the triage prioritization layer.

Agent attribution patterns adopted by the Agentforce UX team for their activity rendering spec. The visual distinction between agent and human entries now appears in their design system guidelines.

"Needs Attention" triage model referenced in the Service Cloud agent notification redesign as prior art for priority-based filtering.

NL querying concept influenced the conversational search pattern adopted by the Einstein Copilot team for record-level questions.

My specific ownership

I owned the interaction model (the attribution-triage-approval loop), all 15+ scenario designs, the NL search behavior, and the production-fidelity prototype. The data architecture (what metadata agents write to the activity record) was owned by our team's architect. Visual design for the insight cards was co-designed with the other designer on the team.

Reflection

What I'd revisit and what's missing

The NL search might be over-scoped

In retrospect, natural language querying is the least validated pillar. Users may not think to ask analytical questions of their timeline. If I restarted, I'd scope the search to task-oriented queries ("find Maria's last email") before expanding to analytical ones ("average sentiment this month").

No user validation

This work was never tested with real users - no AE walkthroughs, no usability studies, no diary studies of timeline usage patterns. The design is grounded in platform knowledge and architectural analysis, not observed behavior. That's a meaningful gap for a project claiming to solve user problems.

Success for vision work

This project's success metric isn't "shipped feature." It's: did it set direction, did it produce reusable patterns, did it de-risk the technical conversation. By those measures, the patent filing and pattern adoption downstream validate the approach - even without production deployment.

Vision work isn't speculative if it changes how teams think about the problem. The patterns designed here are now part of how Salesforce approaches agent-generated content in shared components.
